# CPACodexKeeper 通知整合与降噪需求

## Problem Frame

生产环境中，`CPACodexKeeper 轮次变更通知` 可能每半小时发送一次，飞书被当成运行流水账使用，容易掩盖真正需要处理的告警。同时，账号状态日报与额度日报分开发送，内容中都包含账号数量和可用状态，存在重复。

目标是把飞书通知改成“告警优先 + 定时播报”：实时通知只保留需要人关注的事件；普通状态通过多时段定时播报统一呈现。系统还需要支持多台机器部署，通过配置展示当前服务器名称。

## Requirements

**通知身份与标题**
- R1. 所有飞书通知标题都必须包含可配置的服务器名称，便于多台机器部署时区分来源。
- R2. 服务器名称应可通过配置修改；未配置时使用安全的默认名称。

**实时高风险账号通知**
- R3. 本轮只要有 token 被删除，就发送一条实时删除通知。
- R4. 同一轮多个删除必须合并到一条通知里，不再逐 token 发送多条删除通知。
- R5. 本轮只要有 token 被禁用，就发送一条实时禁用通知。
- R6. 同一轮多个禁用必须合并到一条通知里。
- R7. token 启用不实时通知，只进入定时播报。
- R8. token 刷新不实时通知，只进入定时播报。

**实时额度与异常通知**
- R9. Plus 额度告警继续实时发送，触发条件沿用当前阈值语义：当前可用数过低、5h 平均剩余过低、7d 平均剩余过低。
- R10. Plus 额度恢复继续实时发送。
- R11. CPA API 异常、大面积 usage 检测失败、连续多轮网络失败、网络检测恢复、巡检异常、进程异常退出继续实时发送。

**定时播报**
- R12. 账号状态日报和额度日报合并为一类通知，命名为“定时播报”，不再使用“日报”作为主概念。
- R13. 定时播报支持一天多个发送时段。
- R14. 定时播报默认使用东八区时区语义。
- R15. 定时播报必须包含账号巡检统计：总计、存活、删除、禁用、启用、刷新、跳过、网络失败。
- R16. 定时播报必须包含 quota 概况：Plus / Free 总数、当前可用数、平均剩余额度、exhausted 数、最早重置时间等当前 quota 关键指标。
- R17. 定时播报应包含本轮删除/禁用名单摘要；名单过长时允许截断，避免飞书消息过长。

**降噪边界**
- R18. 普通轮次变更不再每半小时实时推送；只有删除、禁用、额度告警/恢复或异常类事件触发实时通知。
- R19. 启用、刷新等正常维护动作只在定时播报里展示，不单独打扰。
- R20. 受控测试通知仍应保留，用于上线前验证飞书 transport 与消息格式。

## Success Criteria

- 生产环境不再因为普通启用/刷新或普通轮次统计每半小时发送 `CPACodexKeeper 轮次变更通知`。
- 本轮发生删除时，只收到一条合并删除通知。
- 本轮发生禁用时，只收到一条合并禁用通知。
- 启用和刷新不会触发实时通知，但会出现在下一次定时播报中。
- Plus 额度告警和恢复仍可实时到达。
- 定时播报一条消息同时覆盖账号状态和平均额度，不再分别发送账号日报与额度日报。
- 多台服务器部署时，飞书通知标题能清楚显示来源服务器。

## Scope Boundaries

- 不改变 token 删除、禁用、启用、刷新本身的业务决策规则。
- 不新增外部通知渠道；仍使用现有飞书 webhook transport。
- 不把飞书变成完整审计日志；详细流水继续依赖容器日志。
- 不要求定时播报包含每个 token 的完整明细，只保留摘要和必要名单。

## Key Decisions

- 告警优先而非流水账：飞书主要用于需要处理的事件，普通状态进入定时播报。
- 删除实时通知：删除是高风险动作，应立刻可见。
- 禁用实时通知：禁用影响账号池可用性，应立刻可见。
- 启用与刷新不实时通知：它们通常是恢复或维护动作，进入定时播报即可。
- 账号状态与额度概况合并：quota 统计已包含账号数量和可用状态，继续分开发送会重复。
- 使用“定时播报”命名：该通知支持多时段，不适合继续称为日报。
- 默认东八区：符合当前使用习惯，减少 UTC 配置理解成本。

## Dependencies / Assumptions

- 当前代码已有账号维护事件、quota 聚合、飞书发送、独立 quota 状态文件等基础能力。
- 计划阶段需要确认旧配置项如何兼容迁移，避免线上 `.env` 升级后静默失效。
- 计划阶段需要确认测试通知是否扩展为新的合并消息样例。
